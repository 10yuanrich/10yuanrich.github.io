Gemini 2.5：推理预算、长上下文与 Agent 工作流

文档类型：技术报告（Google，2025）
主题：主流模型技术报告、推理模型、工具使用、Agent

解决什么问题

Agent 既要处理代码、文本、图像、音频和视频，又要在复杂任务上投入推理、在常规任务上控制延迟与成本。Gemini 2.5 报告将这一矛盾表述为一条能力-成本 Pareto 前沿：2.5 Pro 承担最强推理与编码，2.5 Flash 提供可控的混合推理，2.0 Flash 和 Flash-Lite 覆盖低延迟、高规模路径。

原作者的论证与方法

作者把 Gemini 2.X 定义为原生多模态、支持超过 100 万 token 长上下文与原生工具调用的一组模型。2.5 Pro 是 thinking model，2.5 Flash 是可控制 thinking budget 的混合推理模型；2.0 Flash 与 Flash-Lite 则是非 thinking 的快速模型。报告称 Pro 在前沿编码与推理基准领先，且可处理至三小时视频；核心方法论不是“一个模型总是更强”，而是在不同预算下调度不同模型与推理深度。

核心机制

  1. 可控推理预算：将质量、成本和延迟变成显式取舍，适合把难题升级处理、把批量任务留在快速路径。
  2. 长上下文加原生工具：模型可在大量多模态资料与代码库中理解任务，并通过 function call 调用搜索、执行或其他外部工具。
  3. Agent 工作流组合：长上下文提供任务状态和证据，推理负责计划与纠错，工具负责改变外部世界；三者缺一不可。

实验与适用场景

报告列出代码、推理、多模态和代理式任务，并用 interactive web app、代码库理解及视频处理说明场景。表格还给出各版本输入输出长度、是否 thinking、是否支持工具的发布时状态。实际适合复杂代码库分析、长视频资料审阅、研究型检索和可验证的多步自动化。应当根据任务配置最大思考预算、最大工具步数和失败回退，而不是让所有任务无限制“想更久”。

整理者归纳

Gemini 2.5 说明主流模型的竞争单位已经从单个 checkpoint 变为可调度的能力族。建立 Agent 时，模型选择器要读懂任务风险：高影响但可验证的任务可使用更深推理并要求证据；低价值的批处理优先 Flash；不可逆动作必须在工具层做权限与确认。这样“推理能力”才会转化为稳定的端到端产出，而非不受控的长轨迹。

限制

报告的能力、上下文、工具支持和 preview 标签以发布日期为准，后续 API 行为可能变化。benchmark 的领先不能直接转为长期自主成功率；工具调用增加了提示注入、权限滥用、状态不同步和成本失控风险。不同模型的输入输出模态也不完全相同，不能用家族名称掩盖差异。

与现有概念的关系

  • Gemini 1.5：百万级上下文如何成为可用工作记忆：2.5 延续长上下文，但把它与 thinking 和工具调用合成 Agent 基础。
  • GPT-5：路由式统一系统与安全完成策略：两者都将能力-成本取舍交给运行时调度。
  • Harness Engineering：模型家族只能提供候选能力，任务状态、验证器、权限和重试策略仍属于 harness。
