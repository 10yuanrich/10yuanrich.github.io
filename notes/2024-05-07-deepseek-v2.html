DeepSeek-V2：以 MLA 和细粒度 MoE 降低训练与推理成本

主题：MoE、KV cache、MLA、专家路由、长上下文

对应材料：DeepSeek-AI，DeepSeek-V2: A Strong, Economical, and Efficient Mixture-of-Experts Language Model（arXiv:2405.04434，2024）

这篇报告解决了什么问题

密集模型扩大容量时，每个 token 都要经过全部 FFN 参数；生成时每层还要缓存所有历史 token 的 KV。前者拉高训练 FLOPs，后者在长上下文和高并发时拉高显存。V2 想同时处理这两类成本，而不是用更低成本换取明显能力损失。

原作者的论证、方法与核心机制

报告提出两个互补结构。第一是 MLA：先把 Key 与 Value 联合低秩压缩为潜向量，解码时只缓存该潜向量，再在计算注意力时上投影。为让 RoPE 不破坏这种吸收关系，论文将位置相关部分与内容部分解耦。它的目标是比 GQA/MQA 更大幅缩小缓存，同时保持比直接共享 KV 更强的表示能力。

第二是 DeepSeekMoE。FFN 被拆为细粒度专家，除被 router 选中的专家外还设共享专家；这样一部分通用知识不必每次经过稀疏路由，专用专家有更强的分工空间。训练时专家并行会使 token 跨设备 all-to-all，论文因此加上 device-limited routing、辅助负载均衡和 token dropping，以控制通信和过载。注意：这些是成套的系统设计，而不是“只要加 MoE 就更便宜”。

模型总参数 236B、每 token 激活 21B，预训练 8.1T token，支持 128K 上下文；之后依次用 SFT 与 GRPO 做对话后训练。

实验或适用场景

相对其 67B 密集前代，原作者报告 V2 节省 42.5% 训练成本、KV cache 降低 93.3%、最大生成吞吐提高至 5.76 倍，并在中英基准和开放式评测中比较 SFT/RL 版本。最直接的适用场景是长会话、长文档或高并发生成服务：缓存与每 token 激活量会比“总参数”更接近实际成本。

整理者归纳

V2 给出一个实用的成本拆分：MoE 主要节省每 token FFN 计算，MLA 主要节省自回归生成状态。两者解决不同瓶颈。因此评估模型部署时，应分开测预填充吞吐、decode 吞吐、KV cache、路由负载和跨卡通信，不能只比较参数量或离线 benchmark。

限制

MLA 的压缩是近似表示，不保证所有任务或所有长度下无损；MoE 的路由、容量和通信对硬件拓扑敏感。报告的百分比相对于特定 67B 基线和实现，不能直接移植到其他服务栈。技术报告没有公开完整预训练数据和所有工程细节，外部复现应以端到端测量验证。

与现有概念的关系

  • Transformer：用自注意力重写序列建模：MLA 保留注意力的读取机制，但压缩了需要跨生成步保存的状态。
  • DeepSeek-V3：把 MoE 的模型能力、训练稳定性和部署成本一起设计：V3 继承 MLA 与 DeepSeekMoE，并调整负载均衡和训练目标。
  • Scaling Laws：MoE 的总参数与激活参数分离，使“模型规模”不能只用一个数描述。
