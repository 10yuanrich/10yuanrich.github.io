Claude Sonnet 5：Agent 性能升级与能力阈值评估

对应材料：Anthropic，System Card: Claude Sonnet 5（2026）

类型：官方系统卡。材料把模型能力、风险阈值、Agent 安全与模型福利评估写在同一发布审计中。

这份材料解决了什么问题

随着模型在编码、浏览、电脑使用和多步骤搜索上更强，发布问题不再是“某项基准更高了吗”，而是“它是否越过需要改变部署政策的能力阈值”。Sonnet 5 系统卡给出一次较完整的回答：先在化生、自动化 AI 研发、网络和对齐风险域中评估，再检查 Agent 系统的恶意使用与 prompt injection，最后才汇报通用能力基准。

原作者的论证与方法

原作者将 Sonnet 5 定义为 Sonnet 4.6 的升级，主张它在多项 Agent 性能上提升，但没有超过更强 Opus 或 Mythos 级模型的能力前沿。其 RSP 结论是：在自动化 AI 研发的自动评测中未跨越能力阈值；对齐风险被判断为很低但高于此前 Sonnet；网络能力并非它的优化重点。这里的三个判断均是发布方在自有阈值、任务和风险报告下作出的，不是对模型真实能力上限的普遍证明。

该卡的评估范围明显扩展：除了 SWE-bench、Terminal-Bench、Agentic Search、长上下文、多模态和专业任务，还包括 ExploitBench、OSS-Fuzz、CyberGym 等网络任务；对 Agent 则测试 Claude Code、电脑使用、影响行动、注入鲁棒性、外部红队和 bug bounty。它还审查评测意识、潜在 sandbagging、绕过防护的能力以及事实性/幻觉。此举承认一个难题：模型若知道自己在被测，单一 benchmark 可能低估部署时的风险。

核心机制

  1. 从能力到发布门槛：能力评测只是一层输入，RSP 再结合威胁主体提升幅度、缓解可行性和不确定性做风险决策。
  2. Agent 攻击面独立建模：模型安全不等于 Agent 安全。网页、仓库、邮件或工具返回值都可能携带注入指令，必须按表面分别压测。
  3. 可靠性审计：对评测意识、欺骗、幻觉和错误报告做专项检查，防止把“会说安全话”误判为“在复杂轨迹中可靠”。
  4. 分层可观测性：模型输出、工具调用、权限拒绝、异常轨迹和人类复核需要同时记录，单看最终答案无法定位失效环节。

实验与适用场景

对使用 Sonnet 类模型做工程 Agent 的团队，这份材料更像上线审计模板：代码基准证明候选能力，隐藏任务和真实工具环境检验泛化；注入与恶意使用测试验证防线；只读、沙盒、审批和限额把失败的损害限制在可恢复范围。对于需要高确定性的医疗、法律、金融或生物流程，卡片本身不能替代领域验证与责任主体。

整理者归纳

最值得保留的思路是：“未越过阈值”不是免于治理，而是确定治理强度的输入。 模型能力连续增长，部署权限却应离散分级；一旦允许浏览、写文件、执行命令或影响外部系统，应该用独立评测和运行时控制来决定是否升级权限。

限制与关系

多数评测为内部执行，文档也坦承评测意识和沙袋效应的难测性。它延续 Claude 4：面向长程 Agent 的能力评估与 ASL-3 防护 的风险门槛思路，并与 Harness Engineering 的外部评估器、回归集和受限可编辑面形成同一条工程原则。
