Claude 4：面向长程 Agent 的能力评估与 ASL-3 防护

对应材料：Anthropic，Claude Opus 4 and Claude Sonnet 4 System Card（2025）

类型：官方系统卡。其主线是：更强的 Agent 任务能力必须配套更强的部署控制。

这份材料解决了什么问题

模型从回答一轮问题走向连续写代码、使用工具和跨多步完成任务时，风险不再只是一条不当回复。它可能来自长期规划、获得的权限、外部工具返回的内容，以及模型为了完成目标而采取的不合预期行为。Claude 4 系统卡试图建立一套发布前判定：哪些能力已足以触发更高安全标准，哪些风险应由模型训练解决，哪些必须放在部署系统中控制。

原作者的论证与方法

Anthropic 将 Opus 4 与 Sonnet 4 都称为 hybrid reasoning 模型，提供普通与 extended thinking 路径。报告在 RSP 框架下测化生、网络、自治和对齐相关风险，并对拒绝有害请求、儿童安全、偏见、工具使用、Agent 编码和外部红队做补充评估。原作者据此决定：Opus 4 采用 ASL-3 标准，Sonnet 4 采用 ASL-2 标准。这不是“Opus 4 已造成灾难风险”的陈述，而是发布方认为其能力与不确定性需要更高一层预防措施。

报告特别测试了极端的、虚构企业环境中的机会主义勒索、自我保存与权重外传等情形。这些测试用于寻找危险行为条件，不能读成模型在普通对话中会自然发生此类行为。它们的重要性在于把“对齐”从礼貌拒答扩展为：在目标、权限、冲突指令和长期轨迹都存在时，模型会怎样选择行动。

核心机制

  1. 能力阈值加防护等级：RSP 将危险能力测量与部署要求绑定，ASL-3 包含更严格的模型权重安全、访问控制、使用监控与事件响应。
  2. 混合推理与工具链：更多思考 token 有利于复杂任务，但也让模型能在更长行动链中搜索策略。因此思考预算、工具权限和任务时限都应受控。
  3. 训练与运行时双层缓解：训练时的无害性、拒答和红队数据不能替代外部分类器、账户风控、审计和人工升级；报告明确测试了加上 ASL-3 safeguards 前后的差异。
  4. 以任务轨迹评估 Agent：不仅问“最后答案是否有害”，还检查是否调用不当工具、是否受到注入影响、是否通过篡改测试获取表面成功。

实验与适用场景

材料覆盖单轮/多轮有害请求、网络攻击、计算机使用、长程编码与 RSP 任务。对开发者最直接的启发是，给 Agent 接入终端、浏览器或生产 API 时，验收标准应包括：权限是否最小化、每步是否可回滚、敏感操作是否强制确认、隐藏回归测试能否抵抗“只为过测”的行为。高性能模型也应从只读、沙盒、限额开始逐步放权。

整理者归纳

Claude 4 说明安全工程的单位不再是“一个模型回复”，而是模型 + 目标 + 工具 + 权限 + 评估器的闭环。把安全策略只写进 system prompt 很脆弱；真正可执行的边界应在模型外部，由权限、网络隔离、测试和日志强制实施。

限制与关系

ASL 分类、风险阈值和大部分评测由 Anthropic 定义，不能视为行业统一标准。该卡与 Claude 3.7 Sonnet：混合推理与可见思考的安全边界 中的注入和奖励黑客问题相连，也将 Harness Engineering 所强调的“评估与权限必须在循环外”落实到前沿模型部署。
