GPT-3：上下文学习如何改变任务适配方式

主题：few-shot、in-context learning、提示、规模化语言模型

对应论文：Tom B. Brown 等，Language Models are Few-Shot Learners（2020）

这篇论文解决了什么问题

传统迁移的默认路径是“拿到任务数据，再更新参数”。这会带来标注、训练、部署版本和每项任务单独维护的成本。GPT-3 系统检验另一条路径：不做梯度更新，只将任务描述及少量输入输出示例放入上下文，大规模自回归模型是否能在一次前向推理中完成新任务？论文将其称为 in-context learning，并区分 zero-shot、one-shot 与 few-shot。

原作者的论证与方法

作者训练了最大 1750 亿参数的自回归 Transformer，并在二十多个 NLP 数据集上比较三种设定：zero-shot 只有自然语言说明，one-shot 加一个示例，few-shot 加多个示例。每种设定都没有下游梯度更新或任务专用微调；模型通过续写或候选答案概率完成预测。

原文报告，多数任务上随模型规模增加，few-shot 增益更明显；例如在 TriviaQA 上，zero/one/few-shot 准确率依次改善，部分结果接近或达到当时的微调系统。论文也明确列出失败任务，并专门检查 Common Crawl 类训练语料可能包含测试集的 data contamination。作者的结论是污染对大部分所测结果影响有限，但这不是对所有 benchmark 或未来网页语料的免检证明。

核心机制：上下文是一次性的任务程序

GPT-3 不修改权重。示例把“输入长什么样、输出采用什么格式、判定边界大致在哪里”编码为 token 序列，模型根据预训练形成的模式在当前窗口内继续生成。因此它的“学习”发生在激活和注意力状态中，而非持久参数中；清空上下文后，这项临时适配也消失。

这解释了 few-shot 的工程性质：示例选择、顺序、标签词、分隔符和 token 预算都会影响结果。它也是 提示工程 的理论起点。将上下文称为“任务程序”是整理者的工作性比喻，不应误读为论文证明模型执行了显式符号程序。

实验与适用场景

适合任务快速变化、示例少、不能或不值得为每项任务训练一个模型的场景，例如原型分类、格式转换、少量示例抽取和交互式分析。先建立 zero/few-shot 基线，能快速判断任务是否可被自然语言接口覆盖。

不适合直接作为高一致性生产承诺。遇到重复量大、输出必须稳定、领域数据敏感或有明确标签时，SFT、InstructGPT：SFT 如何让预训练模型开始遵循指令、检索与程序校验往往比堆叠示例更可控。每次 API 调用都塞入示例还会增加上下文成本和版本漂移风险。

整理者归纳

GPT-3 改变的不是“模型免训练”，而是适配资源的分配：一部分本应写进微调数据和权重的意图，被转移到运行时上下文。这个能力使 prompt 成为产品接口，同时也把评测从“模型固定不变”扩展为“模型、模板、示例和顺序的联合系统”。因此，知识库或应用应版本化保存 prompt、few-shot 例子与评测集，而不能只记录模型名。

限制与容易误解

  • few-shot 并非无数据；它依赖任务说明和高质量示例，只是不更新参数。
  • 能从上下文模仿格式，不等于具备稳定的因果推理或事实检索能力。
  • 175B 的结果不能线性外推到更小模型、不同语料或不同窗口长度。
  • 原文已指出基准污染、偏见和错误；使用时必须做领域验证和风险控制。

与现有概念的关系

  • GPT-2：规模化语言模型的零样本迁移信号：GPT-2 给出规模化零样本信号，GPT-3 将其扩展为系统的 few-shot 评测。
  • GPT-1：预训练加微调如何成为通用 NLP 迁移范式：两者分别代表参数更新和上下文示例两种适配路径。
  • Scaling Laws：GPT-3 的实验依赖能力随规模变化的假设，但具体配比须结合 Kaplan 与 Chinchilla 的证据。
