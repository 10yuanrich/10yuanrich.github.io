RAG：让生成模型从外部知识库取证再回答

主题：检索增强生成、参数化记忆、非参数化记忆、开放域问答

对应论文：Patrick Lewis 等，Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks（2020）

解决什么问题

仅靠预训练参数回答事实问题有三个结构性缺口：知识更新要重训，答案难显示依据，且模型可能把不确定内容流畅地补全。RAG 将知识拆为两部分：seq2seq 模型的参数化记忆负责理解和生成，外部向量索引的非参数化记忆负责按问题取回文档。它要解决的是让生成式系统也能使用显式、可替换的知识源。

原作者的方法与论证

论文使用 BART 作为生成器，使用 DPR 双编码器为问题与文档编码，在 Wikipedia passage 向量索引中取 top-K 文档。文档 z 不是一个固定上下文，而是潜变量：训练目标对候选文档的概率和给定文档生成答案的概率做边缘化。检索器可由 DPR 初始化并在任务上更新。

作者提出两种分解：RAG-Sequence 假设整条输出共用同一检索文档；RAG-Token 允许不同输出 token 依赖不同文档。前者更像“选定证据后写作”，后者更灵活但推理近似与计算更复杂。两者都不是简单把检索文本硬拼给 prompt，而是在生成概率模型中显式结合检索分布。

核心机制

问题 -> DPR 召回 top-K passages -> BART 条件生成 -> 对候选文档概率边缘化。系统同时拥有参数化语言能力和可替换文档索引；Sequence 与 Token 的差别在于一段输出是否共享同一潜在证据文档。

实验与适用场景

论文在 Natural Questions、WebQuestions、TriviaQA、CuratedTREC 等开放域 QA，MS MARCO 生成式回答、Jeopardy 问题生成和 FEVER 上评估。作者报告 RAG 在四个开放域 QA 基准达到当时强结果，并显示相较仅参数化 BART，答案更具多样性；替换不同版本 Wikipedia 索引会改变输出，说明知识确实存在索引侧。

适合有明确文档语料、知识会更新、需要回答可回链的问答、企业知识库与研究辅助。落地时最重要的不是“是否叫 RAG”，而是语料边界、切块、召回、重排、生成约束和引用呈现能否形成可审计链路。

整理者归纳

RAG 最持久的架构价值是把错误分层：先问“没检到”还是“读错/编错”。这比把所有失败都归于模型幻觉更可操作。检索不是事实担保，生成器仍可能误读、混合多篇文档或无视证据；反过来，回答正确也不代表检索证据充分。因此生产系统应记录 query、候选、排序、实际注入片段与最终引用。

限制与关系

  • 原文索引是旧版英文 Wikipedia，覆盖、时效和领域适配均受限；检索不到时生成器仍会回答。
  • RAG-Token 的“不同 token 取不同文档”是模型表述，不应等同于现代 agent 的多轮搜索工作流。
  • 与 REALM：检索感知的语言模型预训练：REALM 把检索纳入预训练，RAG 主要给出通用生成任务的微调配方。
  • 与 Mamba：用选择性状态空间模型线性处理长序列：RAG 用外部可寻址文档扩充知识，Mamba 用压缩状态传递历史，二者解决不同访问瓶颈。
