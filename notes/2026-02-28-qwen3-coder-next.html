Qwen3-Coder-Next：用可执行环境训练小激活参数的代码代理

主题：Qwen、代码代理、MoE、强化学习

对应材料：Qwen Team，M33 Qwen3-Coder-Next Technical Report（arXiv:2603.00729，2026）

这篇报告解决了什么问题

代码代理既需要跨步骤规划和编辑，又受推理成本限制。若仅靠静态代码语料，模型难以学习“运行失败后怎样修复”；若只扩大模型，部署成本又会快速上升。

原作者的论证、方法与核心机制

报告提出总参数 80B、推理仅激活 3B 的开源代码模型，并通过大规模合成的可验证编码任务和可执行环境，在 mid-training 与强化学习阶段直接从环境反馈学习。

稀疏专家把容量与每 token 的计算量分离；可执行任务将奖励从偏好标签变为测试、编译或终端状态。训练目标因此不仅约束答案文本，还约束模型能否在工具环境里产生可验证的状态变化。

实验或适用场景

作者在 SWE-Bench、Terminal-Bench 等代理型基准上报告其相对活跃参数规模的竞争表现，并发布 base 与 instruction 版本。这里的“高效”应以端到端工具循环吞吐而非单次 token 速度衡量。

整理者归纳

它给出的实用方向是：代码代理的训练数据价值来自可验证环境。企业落地可以先把单元测试、lint、构建、沙箱和回滚做成可靠反馈，再评估是否值得引入更长的代理轨迹或 RL；没有反馈闭环，模型再大也很难稳定自修正。

限制

可验证任务会偏向有测试的目标，需求理解、架构权衡和无测试遗留系统仍难自动衡量；小激活参数也不意味着低总显存或无通信成本。

与现有概念的关系

  • 与 Qwen2.5-Coder：把代码能力训练到仓库级编辑与工具环境里 相连：两者都从代码生成走向环境交互，Coder-Next 更明确以可验证任务和 RL 作为训练信号。
  • Transformer：用自注意力重写序列建模：所有这些报告仍以 Transformer 系列为基础；实际差异主要落在数据、后训练、稀疏化、模态编码和服务工作流。
