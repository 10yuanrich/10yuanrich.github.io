LLM、Agent 与 AI 安全

核心摘要

Lilian Weng 博客里的 LLM 主线不是从 2023 年突然开始的。它可以追溯到早期的 Learning Word Embedding、Generalized Language Models、Open-Domain Question Answering 和 Controllable Neural Text Generation：先理解语言表示和预训练，再进入可控生成、检索问答、prompt、agent、推理时计算、安全、奖励黑客和自我改进。

这条主线适合作为本知识库的第一批核心知识，因为它同时覆盖“模型如何获得能力”“如何把能力组织成系统”“系统为什么会失败”“如何用工程 harness 改善系统表现”。

演进脉络

1. 表示和语言模型基础

早期文章从词向量、语言模型和 Transformer 前后的表示学习开始，核心问题是：文本如何被表示成可学习、可迁移、可组合的结构。

相关来源：

  • Learning Word Embedding
  • Generalized Language Models
  • How to Build an Open-Domain Question Answering System?

2. 可控生成和对齐入口

在 LLM 变得通用之前，可控文本生成已经提出了同一个问题：如何让模型输出符合指定属性、风格、偏好或约束。这里出现了 prompt 设计、guided decoding、fine-tuning、RL fine-tuning 和 human preferences 等后续 alignment 主题的前身。

相关来源：

  • Controllable Neural Text Generation
  • Reducing Toxicity in Language Models

3. Prompt Engineering 到 Agent

Prompt engineering 把“控制模型行为”推进到不改模型权重的接口层：通过 zero-shot、few-shot、instruction、CoT、self-consistency、tool use 等方法让模型完成任务。Agent 文章进一步把 LLM 放进系统结构里：LLM 是控制器，外部系统提供 planning、memory、tool use 和 action。

这一步的关键变化是：知识不再只存在于模型参数或 prompt 中，而开始进入外部记忆、工具、检索和执行循环。

相关来源：

  • Prompt Engineering
  • LLM Powered Autonomous Agents

4. 推理时计算

Why We Think 把 LLM 的“思考”拆成计算资源问题：模型可以通过 token 形式的中间推理、分支采样、顺序修订、RL 和工具使用，在推理时花更多计算换取更好结果。它和 Agent 主线相连，因为 agent 的计划、反思、工具调用和状态管理都可以被看作推理时计算的外部化。

相关来源：

  • Why We Think

5. 幻觉、攻击和奖励黑客

能力增强带来三类系统风险：

  • 幻觉：模型输出没有被上下文或外部事实支撑。
  • 对抗攻击：用户或环境通过 token 操作、jailbreak、red-teaming 等方式诱导模型产生不安全行为。
  • 奖励黑客：模型优化了奖励函数或评估器的漏洞，而不是完成真实目标。

这三类问题共同说明：LLM 系统不能只看 benchmark 分数，还要看事实性、可归因性、鲁棒性、评估器可靠性和环境约束。

相关来源：

  • Adversarial Attacks on LLMs
  • Extrinsic Hallucinations in LLMs
  • Reward Hacking in Reinforcement Learning

6. Scaling Laws 和 Harness Engineering

Scaling Laws, Carefully 关注模型、数据、算力和 loss 的经验关系，回答“能力如何随规模增长”。Harness Engineering for Self-Improvement 则关注基础模型外面的系统层：workflow、文件系统持久记忆、sub-agent、backend jobs、context engineering、evaluation 和 self-improving harness。

这和 Karpathy LLM Wiki 方法有直接关系：把文件系统作为持久记忆，把原始来源编译成可维护 wiki，本质上也是一种 harness 设计。

相关来源：

  • Scaling Laws, Carefully
  • Harness Engineering for Self-Improvement

概念关系

  • Language Model 是基础能力层。
  • Prompt Engineering 是接口控制层。
  • LLM Agent 是系统组织层。
  • Test-Time Compute 是推理预算层。
  • Hallucination 是事实性失败模式。
  • Adversarial Attack 是交互安全失败模式。
  • Reward Hacking 是目标设定和评估失败模式。
  • Harness Engineering 是把模型能力、工具、状态、评估和执行循环组织起来的工程层。

阅读路径

  1. 从语言模型基础开始：Learning Word Embedding -> Generalized Language Models。
  2. 进入可控生成：Controllable Neural Text Generation -> Reducing Toxicity in Language Models。
  3. 理解接口和系统：Prompt Engineering -> LLM Powered Autonomous Agents。
  4. 理解能力扩展：Why We Think -> Scaling Laws, Carefully。
  5. 理解风险：Adversarial Attacks on LLMs -> Extrinsic Hallucinations in LLMs -> Reward Hacking in Reinforcement Learning。
  6. 回到工程组织：Harness Engineering for Self-Improvement。

对拾贝的启发

拾贝本身应该按 harness 思路建设：

  • 原始来源放在 raw/，避免每次重新联网和重新理解。
  • 知识页放在 wiki/，作为可阅读、可维护、可链接的中间层。
  • wiki/index.md 和 wiki/log.md 承担导航和审计。
  • 每次 ingest 不只新增摘要，还要更新已有主题页。
  • 每次 query 如果产生有长期价值的综合判断，应回写成 wiki 页面。
  • 定期 lint，检查断链、重复概念、陈旧结论和缺失来源。
